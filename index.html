    用一个栈作为压入栈，在压入数据时只往这个栈中压入，记为stackPush；另一个栈只作为弹出栈，在弹出数据时只从这个栈弹出，记为stackPop。

队列是先进先出的，而数据压入栈的时候，顺序是先进后出的，那么只要把stackPush中的数据弹出再压入stackPop中，顺序就变回来了。

不过需要注意两点：

  1. 如果stackPush需要往stackPop中压入数据，那么必须一次性把stackPush中的数据全部弹出和压入stackPop。
  2. 如果stackPop不为空，那么stackPush绝对不能向stackPop中压入数据，否则导致顺序错乱。

有getMin功能的栈

发表于 2017-05-10 | 分类于 算法

要求：

  1. pop、push、getMin操作的时间复杂度都是O(1)。
  2. 设计的栈类型可以使用现成的栈结构。

第一种解法

在设计上使用两个栈，以空间换时间，一个栈用来保存当前栈中的元素，其功能和一个正常的栈没有区别，这个栈记为stackData；另一个栈用于保存每一次pop或push之后的最小值，记为stackMin。

压入数据规则

假设当前数据为newNum，先将其压入stackData，然后判断stackMin是否为空：

=>如果为空，则newNum也压入stackMin。
=>如果不为空，则比较newNum和stackMin的栈顶元素的大小。
==>如果newNum小于等于stackMin的栈顶元素，则newNum也压入stackMin。
==>如果newNum大于stackMin的栈顶元素，stackMin不压入任何内容。

弹出数据规则

先在stackData中弹出栈顶元素，记为value，然后比较当前stackMin的栈顶元素和value哪一个更小。

通过压入规则可知，stackMin中存在的元素是从栈底到栈顶逐渐变小的，stackMin栈顶的元素既是当前stackMin栈的最小值，也是当前stackData栈的最小值。所以value只可能大于或者等于stackMin的栈顶元素。

=>如果value等于stackMin栈顶元素，那么stackMin弹出栈顶元素。
=>如果value大于stackMin栈顶元素，那么stackMin不弹出栈顶元素，返回value。

所有说，每一次pop和push之后stackMin中的栈顶元素始终是当前stackData中的最小值。

压入数据规则

假设当前数据为newNum，先将其压入stackData，然后判断stackMin是否为空：

一、什么是 npm 脚本？
npm 允许在 package.json 文件里面，使用 scripts 字段定义脚本命令。

上面代码是 package.json 文件的一个片段，里面的 scripts 字段是一个对象。它的每一个属性，对应一段脚本。比如，build 命令对应的脚本是 node build.js。
命令行下使用 npm run 命令，就可以执行这段脚本。

项目的相关脚本，可以集中在一个地方。
不同项目的脚本命令，只要功能相同，就可以有同样的对外接口。用户不需要知道怎么测试你的项目，只要运行 npm run test 即可。

查看当前项目的所有 npm 脚本命令，可以使用不带任何参数的 npm run 命令。

二、原理
npm 脚本的原理非常简单。每当执行 npm run，就会自动新建一个 Shell，在这个 Shell 里面执行指定的脚本命令。因此，只要是 Shell（一般是 Bash）可以运行的命令，就可以写在 npm 脚本里面。
比较特别的是，npm run 新建的这个 Shell，会将当前目录的 node_modules/.bin 子目录加入 PATH 变量，执行结束后，再将 PATH 变量恢复原样。
这意味着，当前目录的 node_modules/.bin 子目录里面的所有脚本，都可以直接用脚本名调用，而不必加上路径。比如，当前项目的依赖里面有 Mocha，只要直接写 mocha test 就可以了。

由于 npm 脚本的唯一要求就是可以在 Shell 执行，因此它不一定是 Node 脚本，任何可执行文件都可以写在里面。
npm 脚本的退出码，也遵守 Shell 脚本规则。如果退出码不是0，npm 就认为这个脚本执行失败。

页面的渲染过程和生命周期钩子

发表于 2017-05-09 | 分类于 HTML

通俗的说，浏览器就是一个展示网页的应用程序，一个网页的主要组件就是描述内容的 html 标签及网页相关的外部资源，比如图片、脚本、样式等。浏览器的工作就是将这些文本化的 html 标签转变为图形展示给用户。

每一个浏览器都用不同的方法去完成这个任务。但都遵循一个共同的处理流程，如下图：

浏览器第一件事情是下载指定的页面，当我们在输入完一个网址后，浏览器的检索引擎通过网址 url 去定位及下载这个页面，并将这个页面保存到资源缓存中，如果这个页面存在于缓存中，它将跳过下载，而是调用缓存的数据。

当检索引擎完成一个页面的下载，浏览器会快速的解析 html 标签，并发现跟页面相关的外部资源，如图片，css 或者其它的多媒体。检索引擎然后在去下载这些外部资源。

检索引擎下载外部资源的时候，浏览器开始解析网页的 html 标签，并建立一个 DOM 树。此时，基于文本的 html 标签变成了可以被浏览器操控的对象。

浏览器之后开始解析 css 样式及定义在 html 标签相关联的样式，并且创建一个样式对象，styleobject。这个样式对象被用来确定浏览器如何显示一个网页。

当有了 DOM tree 及样式对象后，浏览器开始建立一个渲染树。渲染树有的时候跟 DOM 树是有不同的，因为有的 DOM 项因为 css 设置为不显示。那么这些项将不会出现在渲染树里面，而只会在 DOM 里面。并且这些项不会出现在浏览器的下一步处理中。

现在浏览器开始 flow，这个过程是由浏览器的定位引擎计算每个 DOM 项的大小及相对于整个页面的位置。当这个过程处理完时，所有的 DOM 项被定位好了，最终整个页面的布局也就好了。

最后一步是显示整个页面，这个工作是浏览器的绘画引擎完成，显示整个最终的效果。也可以称为 paint。

此时页面加载完成，触发 load 事件。

可是，整个过程并不是这么简单。当一个页面被渲染完成。浏览器将会进入到事件循环，浏览器开始等待某些事情的发生。我们可以称为 event。每一个事件可能来至用户的点击或者重置浏览器窗口的大小。当浏览器捕获到一个事件，而这个事件是影响整个页面的布局，浏览器会重新做 reflow, repaint 的过程。更新整个页面的布局。reflow 一定会引起 repaint，repaint 不一定会引起 reflow。

当卸载页面的时候，会触发 unload 事件。

em还是rem

发表于 2017-03-17 | 分类于 CSS

首先介绍下这两个单位，em和rem都是相对单位，由浏览器转换为像素值，具体取决于您的设计中的字体大小设置。看下面的例子：

首先你要知道的是每个浏览器有一个默认的font-size，比如Chrome浏览器里默认的font-size为16px，所以这里计算html元素的font-size时1rem=16px，使得html元素的font-size值为64px。

接着当计算body的font-size时，此时的1rem=64px，所以body元素的font-size值为5 * 64 = 320px。

当计算div.wrapper的时候，此时的1rem=64px，html元素作为页面的根元素，rem以它的font-size为基准。所以div.wrapper的font-size为192px。

同理，对于最内层的div来说，1rem=64px，所以font-size为192px，padding-left为128px。

还是同样的例子，只不过单位换成了em。

同样的，这里计算html元素的font-size时1em=16px，使得html元素的font-size值为64px。

接着当计算body的font-size时，根据继承，此时的1em=64px，所以body元素的font-size值为320px。

计算div.wrapper时，根据继承，此时1em=320px，所以div.wrapper的font-size值为960px。

计算最内层的div时，根据继承，此时1em=960px，所以最内层div的font-size值为2880px，注意，由于它的font-size值被显示设置了，所以此时1em=2880px，padding-left的值为5760px。

总结如下：

  1. 每个浏览器有一个默认的font-size，例如Chrome浏览器的为16px。
  2. html元素作为页面的根元素，不管使用em还是rem，都是以浏览器默认的font-size为基准。如果html元素显式的设置了font-size，它的所有的后代元素中，使用rem的都将以它的font-size为基准，否则以浏览器默认的font-size为基准。
  3. font-size具有继承性，如果某个元素某个属性使用em且没有显式设置font-size时，那么会把继承而来的font-size作为基准，否则会以自己的font-size作为基准。
  4. 如果某个元素显式设置了font-size，且单位为em，那么会使用继承值计算出自己的font-size。

理解viewport

发表于 2016-12-27 | 分类于 技术栈

在查阅ppk大神写的几篇关于viewport的文章后，我是这么理解移动端下viewport这么个“磨人的小妖精”的。先上一张图：

结论：visual viewport的大小表示能够“看到”多大的layout viewport，如果它们大小相等，那么透过visual viewport就能“看到”完整的layout viewport；如果visual viewport比layout viewport小，那么只能透过visual viewport“看到”一部分的layout viewport；如果visual viewport比layout viewport大，除了能透过visual viewport“看到”完整的layout viewport，还会包括之外的范围。
拿iphone5为例，官方分辨率参数为1136*640，这是指物理像素，设备像素比为2，所以一个css像素由4个物理像素表示，css屏幕大小为568*320，通过下面的案例理解一下：

根据ppk大神的讲解，layout viewport的宽高可由document.documentElement.clientWidth和document.documentElement.clientHeight得出，在iphone5下是
1739 * 980，这也是大多数机型采用的设置；visual viewport的宽高可由window.innerWidth和window.innerHeight得出，这里是1740 * 980，也就是说上面的案例中layout viewport和visual viewport大小是一样；屏幕大小可由screen.width和screen.height得出，这里是568 * 320。
所有布局都是相对于layout viewport的，所以上面宽度为490的方块应该占据一半的屏幕，结果却是如此。
添加如下标签：
<meta name="viewport" content="width=980, initial-scale=2">
上面标签中的width=980表示设置layout viewport的宽度为980，此时visual viewport的宽度为160，屏幕宽度仍为320。所以initial-scale=2控制的是屏幕和visual viewport中间的比例，用下面的公式表示：
initial-scale = screen.width / window.innerWidth
当width和initial-scale同时出现的时候，取计算之后的最大值。比如width为120，屏幕宽度为320，那么计算之后visual viewport的宽度为160，此时layout viewport的宽度应为160。
在没有设置上面的meta标签的时候，浏览器会自动计算一个默认的initial-scale值，从而使得浏览器不会出现滚动条。比如上面的例子中，visual viewport和layout viewport大小都为980，initial-scale的值为0.33(320/980)左右。
ppk大神认为有三个viewport存在，分别是ideal viewport、visual viewport和layout viewport，其实屏幕就是ideal viewport，它们之间的关系可以总结为：visual viewport表示可以显示多大的layout viewport，之后再缩放成为屏幕大小。
那么如何让三个viewport大小相等，通过下面的标签设置：
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
上面标签的意思如下：让layout viewport的宽度等于屏幕也就是ideal viewport的宽度，通过initial-scale控制visual viewport的宽度等于ideal viewport的宽度，从而让它们三个大小相等。
遇到有趣的问题：我设置了如下标签：
<meta name="viewport" content="initial-scale=0.1">
理想中visual viewport值为3200，没有设置width，所以layout viewport值为默认值980，这里取最大值3200，但是实际上两者都为1280，说明有一个上限。
